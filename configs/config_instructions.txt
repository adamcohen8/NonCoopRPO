Simulation YAML Config Instructions
File: configs/simulation_template.yaml

Purpose
- Define a full simulation setup in one YAML file:
  - rocket
  - chaser
  - target
  - simulator
  - outputs
  - monte_carlo (optional)

Quick Start
1. Copy the template:
   cp configs/simulation_template.yaml configs/my_scenario.yaml
2. Edit your scenario values in configs/my_scenario.yaml.
3. Validate load/parsing:
   .venv/bin/python examples/Simulation_Config_Load_Demo.py --config /Users/adamcohen/Downloads/NonCooperativeRPO/configs/my_scenario.yaml
4. Run the master simulator from config only:
   .venv/bin/python run_simulation.py --config /Users/adamcohen/Downloads/NonCooperativeRPO/configs/my_scenario.yaml

Top-Level Sections
- scenario_name: Human-readable scenario label.
- metadata: Freeform notes/tags.
- rocket: Rocket object settings and algorithm pointers.
- chaser: Chaser object settings and algorithm pointers.
- target: Target object settings and algorithm pointers.
- simulator: Dynamics/environment/time/termination settings.
- outputs: Output behavior (stats/plots/animations/MC outputs).
- monte_carlo: Optional campaign settings and parameter variations.

Algorithm Pointer Format
- Any guidance/orbit_control/attitude_control/mission_objectives entry can use:
  kind: "python"
  module: "python.module.path"
  class_name: "ClassName"
  function: "optional_function_name"
  file: "optional/path/to/file.py"
  params:
    key: value

Notes:
- Use module + class_name for class-based controllers/guidance.
- Use module + function for function-based handlers.
- params is a freeform dict passed later by scenario wiring code.

Mission Module Pointers
- mission_objectives are executed each master_sim step when provided.
- Satellite mission module:
  module: "sim.mission.modules"
  class_name: "SatelliteMissionModule"
  params:
    orbital_mode: "coast" | "pursuit_knowledge" | "evade_knowledge" | "pursuit_blind" | "evade_blind"
    attitude_mode: "hold_eci" | "hold_ric" | "spotlight" | "sun_track" | "pursuit" | "evade" | "sensing"
    target_id: "<object_id>"
    max_accel_km_s2: <float>
    hold_quat_bn: [q0,q1,q2,q3]
    hold_quat_br: [q0,q1,q2,q3]
    boresight_body: [x,y,z]
    spotlight_lat_deg: <float>
    spotlight_lon_deg: <float>
    spotlight_alt_km: <float>
    spotlight_ric_direction: [R,I,C]
    use_knowledge_for_targeting: true|false
- End-state maneuver mission module (generic desired end-state + current knowledge):
  module: "sim.mission.modules"
  class_name: "EndStateManeuverMissionModule"
  params:
    target_id: "<object_id>"
    use_knowledge_for_targeting: true|false
    desired_state_source: "target" | "explicit"
    desired_position_eci_km: [x,y,z]   # required when desired_state_source=explicit
    desired_velocity_eci_km_s: [vx,vy,vz]  # required when desired_state_source=explicit
    strategy: "thrust_limited" | "impulsive"
    max_thrust_n: <float>
    min_thrust_n: <float>
    burn_dt_s: <float>
    available_delta_v_km_s: <float>
    require_attitude_alignment: true|false
    thruster_position_body_m: [x,y,z] | null
    thruster_direction_body: [x,y,z]
    alignment_tolerance_rad: <float>
    terminate_on_velocity_tolerance_km_s: <float>
- Integrated command mission module (mission brain that arbitrates orbital + attitude commands each timestep):
  module: "sim.mission.modules"
  class_name: "IntegratedCommandMissionModule"
  params:
    target_id: "<object_id>"
    desired_state_source: "target" | "explicit"
    use_knowledge_for_targeting: true|false
    desired_position_eci_km: [x,y,z]          # for desired_state_source=explicit
    desired_velocity_eci_km_s: [vx,vy,vz]     # for desired_state_source=explicit
    require_attitude_alignment: true|false
    thruster_direction_body: [x,y,z]
    alignment_tolerance_rad: <float>
    min_burn_accel_km_s2: <float>
- Predictive integrated command mission module (plan at future time, slew early, fire at gate):
  module: "sim.mission.modules"
  class_name: "PredictiveIntegratedCommandMissionModule"
  params:
    target_id: "<object_id>"
    use_knowledge_for_targeting: true|false
    lead_time_s: <float>
    predict_dt_s: <float>
    alignment_tolerance_rad: <float>
    thruster_direction_body: [x,y,z]
    min_burn_accel_km_s2: <float>
    mu_km3_s2: <float>
- Rocket mission module:
  module: "sim.mission.modules"
  class_name: "RocketMissionModule"
  params:
    launch_mode: "go_now" | "go_when_possible" | "wait_optimal_window"
    orbital_goal: "pursuit" | "predefined_orbit"
    target_id: "<object_id>"
    go_when_possible_margin_m_s: <float>
    window_period_s: <float>
    window_open_duration_s: <float>
    predef_target_alt_km: <float>
    predef_target_ecc: <float>

Bridge Pointer Format (SIL/HIL)
- bridge:
    enabled: true|false
    mode: "sil" | "hil"
    endpoint: "udp://host:port" (or your bridge endpoint format)
    module: "bridge.module"
    class_name: "BridgeClass"
    params: {}

Agent Sections (rocket/chaser/target)
- enabled: true/false
- role: descriptive role string
- specs: physical or preset references
- initial_state: initial conditions (or deployment source)
  - satellites support either:
    - Cartesian state: position_eci_km + velocity_eci_km_s
    - COEs via initial_state.coes:
      - a_km (or semi_major_axis_km)
      - ecc (or e)
      - inc_deg (or inclination_deg)
      - raan_deg
      - argp_deg (or arg_periapsis_deg)
      - true_anomaly_deg (or ta_deg)
    - if both are provided, Cartesian position_eci_km/velocity_eci_km_s is used.
  - chaser can also be initialized relative to target in RIC:
    - compact fields:
      - initial_state.relative_ric_rect: [R, I, C, dR, dI, dC]
      - initial_state.relative_ric_curv: [R, I, C, dR, dI, dC]
    - explicit block:
      - initial_state.relative_to_target_ric:
          frame: "rect" | "curv"
          state: [R, I, C, dR, dI, dC]
    - units:
      - position terms in km
      - velocity terms in km/s
    - notes:
      - this is target-centered (chaser wrt target).
      - when present, this relative initialization overrides chaser absolute ECI/COE startup state
        (except rocket_deployment mode).
- guidance: rocket guidance or agent-level guidance pointer
- orbit_control: orbit controller pointer
- attitude_control: attitude controller pointer
- mission_objectives: list of objective pointers
  - these can act as mission modules that drive thrust/attitude/launch timing.
- bridge: SIL/HIL pointer
- knowledge: refresh/conditions/noise/estimation settings

Simulator Section
- scenario_type:
  - auto
  - asat_phased_engagement
  - rocket_ascent
  - free_tumble_one_orbit
  - free_tumble_one_orbit_ric
- duration_s: total simulation duration
- dt_s: primary simulation step
- initial_jd_utc: optional epoch start (null if unused)
- dynamics: orbit/attitude/rocket model switches
  - optional substep controls for satellite propagation:
    - simulator.dynamics.orbit.orbit_substep_s
    - simulator.dynamics.attitude.attitude_substep_s
    - when set, these run orbit/attitude integration at finer internal timesteps than simulator.dt_s.
- environment: ephemeris/atmosphere/etc settings
- plugin_validation:
  - strict: true/false
  - true: fail-fast if any plugin pointer violates contract.
  - false: attempt fallback defaults where supported.
- termination:
  - earth_impact_enabled: true/false
  - earth_radius_km: usually 6378.137

Outputs Section
- output_dir: destination root directory for generated files.
- mode: "interactive" | "save" | "both"
  - interactive: display figures/animations in IDE, do not save by default.
  - save: save outputs to files, no interactive display.
  - both: show and save.
- stats: options for printed and saved statistics.
  Example keys: enabled, print_summary, save_json, save_csv.
- plots: figure output options.
  Example keys: enabled, figure_ids, dpi.
  Optional key: reference_object_id (used for RIC frame trajectory plots).
  Supported figure_ids currently:
  - orbit_eci
  - trajectory_ecef
  - trajectory_ric_rect
  - trajectory_ric_curv
  - trajectory_ric_rect_2d
  - trajectory_ric_curv_2d
  - trajectory_eci_multi
  - trajectory_ecef_multi
  - trajectory_ric_rect_multi
  - trajectory_ric_curv_multi
  - trajectory_ric_rect_2d_multi
  - trajectory_ric_curv_2d_multi
  - attitude
  - quaternion_eci
  - quaternion_ric
  - rates_eci
  - rates_ric
  - relative_range
  - knowledge_timeline
  - control_thrust
  - control_thrust_multi
  - control_thrust_ric
  - control_thrust_ric_multi
  Optional key: ric_2d_planes (list of "ri" | "ic" | "rc")
    - ri: R-I projection with R on y-axis
    - ic: I-C projection with C on y-axis
    - rc: R-C projection with R on y-axis
- animations: animation options.
  Example keys: enabled, types, fps, speed_multiple, draw_earth_map.
  Supported types currently:
  - ground_track: one animation per object
  - ground_track_multi: all objects on one ground-track animation
- monte_carlo: Monte Carlo output options.
  Example keys:
  - save_iteration_summaries: save per-run summary JSON files.
  - save_aggregate_summary: save overall MC summary JSON.
  - save_histograms/display_histograms: existing aggregate histogram plots.
  - save_ops_dashboard/display_ops_dashboard: mission-ops dashboard PNG with thresholds and pass/fail coloring.
  - save_raw_runs: save compact per-run detail JSON.
  - baseline_summary_json: optional path to prior summary JSON for delta comparison.
  - success_termination_reasons: termination reasons that count as success (default includes rocket_orbit_insertion).
  - require_rocket_insertion: require insertion-achieved for run pass.
  - gates: optional pass/fail thresholds.
    - min_closest_approach_km
    - max_duration_s
    - max_total_dv_m_s
    - max_guardrail_events
    - max_total_dv_m_s_by_object: { object_id: max_dv_m_s }

Generated MC artifacts when save_aggregate_summary=true:
- master_monte_carlo_summary.json (full aggregate + runs + reproducibility metadata)
- master_monte_carlo_commander_brief.json
- master_monte_carlo_commander_brief.md
- master_monte_carlo_analyst_pack.json (includes run_details + sensitivity ranking)
- master_monte_carlo_ops_dashboard.png (when save_ops_dashboard=true)
- master_monte_carlo_run_details.json (when save_raw_runs=true)

Monte Carlo Section
- enabled: true/false
- iterations: number of runs
- base_seed: seed offset
- variations: list of parameter variation definitions

Variation Entry Fields
- parameter_path: dot-path to parameter to vary
  Example: "rocket.initial_state.launch_azimuth_deg"
  Example: "chaser.initial_state.deploy_dv_body_m_s[0]"
- mode: "choice" | "uniform" | "normal"
- choice mode:
  - options: [v1, v2, ...]
- uniform mode:
  - low: value
  - high: value
- normal mode:
  - mean: value
  - std: value

Validation Rules (Current Loader)
- YAML root must be a mapping/object.
- simulator.dt_s > 0
- simulator.duration_s > 0
- simulator.scenario_type must be non-empty.
- outputs.mode must be one of: interactive, save, both.
- outputs.output_dir must be non-empty.
- monte_carlo.iterations > 0
- monte_carlo.variations[*].parameter_path must be a non-empty string.

Plugin Contract Validation (strict mode)
- rocket.guidance: must construct and implement command(state, sim_cfg, vehicle_cfg)
- rocket/chaser/target orbit_control: must construct and implement act(belief, t_s, budget_ms)
- rocket/chaser/target attitude_control: must construct and implement act(...)
- bridge (if enabled): must provide at least one of step/start/send_command/receive_command
- mission_objective pointers: must provide a callable function or class method in {evaluate, update, check, act}

Important Current Scope
- The loader validates and parses the YAML into typed config objects.
- Full end-to-end execution from YAML is not yet universal for all scenario runners.
- Use this file now for standardized scenario definitions and input validation.

Recommended Workflow
1. Keep a base scenario YAML.
2. Create variants by changing only metadata + key parameter overrides.
3. Validate with Simulation_Config_Load_Demo.
4. Run your chosen scenario script using parsed config integration as available.
