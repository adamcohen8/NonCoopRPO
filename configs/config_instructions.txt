Simulation YAML Config Instructions
File: configs/simulation_template.yaml

Purpose
- Define a full simulation setup in one YAML file:
  - rocket
  - chaser
  - target
  - simulator
  - outputs
  - monte_carlo (optional)

Quick Start
1. Copy the template:
   cp configs/simulation_template.yaml configs/my_scenario.yaml
2. Edit your scenario values in configs/my_scenario.yaml.
3. Validate load/parsing:
   .venv/bin/python examples/Simulation_Config_Load_Demo.py --config /Users/adamcohen/Downloads/NonCooperativeRPO/configs/my_scenario.yaml
4. Run the master simulator from config only:
   .venv/bin/python run_simulation.py --config /Users/adamcohen/Downloads/NonCooperativeRPO/configs/my_scenario.yaml

Top-Level Sections
- scenario_name: Human-readable scenario label.
- metadata: Freeform notes/tags.
- rocket: Rocket object settings and algorithm pointers.
- chaser: Chaser object settings and algorithm pointers.
- target: Target object settings and algorithm pointers.
- simulator: Dynamics/environment/time/termination settings.
- outputs: Output behavior (stats/plots/animations/MC outputs).
- monte_carlo: Optional campaign settings and parameter variations.

Algorithm Pointer Format
- Any guidance/orbit_control/attitude_control/mission_objectives entry can use:
  kind: "python"
  module: "python.module.path"
  class_name: "ClassName"
  function: "optional_function_name"
  file: "optional/path/to/file.py"
  params:
    key: value

Notes:
- Use module + class_name for class-based controllers/guidance.
- Use module + function for function-based handlers.
- params is a freeform dict passed later by scenario wiring code.

Bridge Pointer Format (SIL/HIL)
- bridge:
    enabled: true|false
    mode: "sil" | "hil"
    endpoint: "udp://host:port" (or your bridge endpoint format)
    module: "bridge.module"
    class_name: "BridgeClass"
    params: {}

Agent Sections (rocket/chaser/target)
- enabled: true/false
- role: descriptive role string
- specs: physical or preset references
- initial_state: initial conditions (or deployment source)
- guidance: rocket guidance or agent-level guidance pointer
- orbit_control: orbit controller pointer
- attitude_control: attitude controller pointer
- mission_objectives: list of objective pointers
- bridge: SIL/HIL pointer
- knowledge: refresh/conditions/noise/estimation settings

Simulator Section
- scenario_type:
  - auto
  - asat_phased_engagement
  - rocket_ascent
  - free_tumble_one_orbit
  - free_tumble_one_orbit_ric
- duration_s: total simulation duration
- dt_s: primary simulation step
- initial_jd_utc: optional epoch start (null if unused)
- dynamics: orbit/attitude/rocket model switches
- environment: ephemeris/atmosphere/etc settings
- plugin_validation:
  - strict: true/false
  - true: fail-fast if any plugin pointer violates contract.
  - false: attempt fallback defaults where supported.
- termination:
  - earth_impact_enabled: true/false
  - earth_radius_km: usually 6378.137

Outputs Section
- output_dir: destination root directory for generated files.
- mode: "interactive" | "save" | "both"
  - interactive: display figures/animations in IDE, do not save by default.
  - save: save outputs to files, no interactive display.
  - both: show and save.
- stats: options for printed and saved statistics.
  Example keys: enabled, print_summary, save_json, save_csv.
- plots: figure output options.
  Example keys: enabled, figure_ids, dpi.
  Optional key: reference_object_id (used for RIC frame trajectory plots).
  Supported figure_ids currently:
  - orbit_eci
  - trajectory_ecef
  - trajectory_ric_rect
  - trajectory_ric_curv
  - trajectory_eci_multi
  - trajectory_ecef_multi
  - trajectory_ric_rect_multi
  - trajectory_ric_curv_multi
  - attitude
  - quaternion_eci
  - quaternion_ric
  - rates_eci
  - rates_ric
  - relative_range
  - knowledge_timeline
  - control_thrust
  - control_thrust_multi
- animations: animation options.
  Example keys: enabled, types, fps, speed_multiple.
- monte_carlo: Monte Carlo output options.
  Example keys: save_iteration_summaries, save_aggregate_summary, save_histograms, save_raw_runs.

Monte Carlo Section
- enabled: true/false
- iterations: number of runs
- base_seed: seed offset
- variations: list of parameter variation definitions

Variation Entry Fields
- parameter_path: dot-path to parameter to vary
  Example: "rocket.initial_state.launch_azimuth_deg"
  Example: "chaser.initial_state.deploy_dv_body_m_s[0]"
- mode: "choice" | "uniform" | "normal"
- choice mode:
  - options: [v1, v2, ...]
- uniform mode:
  - low: value
  - high: value
- normal mode:
  - mean: value
  - std: value

Validation Rules (Current Loader)
- YAML root must be a mapping/object.
- simulator.dt_s > 0
- simulator.duration_s > 0
- simulator.scenario_type must be non-empty.
- outputs.mode must be one of: interactive, save, both.
- outputs.output_dir must be non-empty.
- monte_carlo.iterations > 0
- monte_carlo.variations[*].parameter_path must be a non-empty string.

Plugin Contract Validation (strict mode)
- rocket.guidance: must construct and implement command(state, sim_cfg, vehicle_cfg)
- rocket/chaser/target orbit_control: must construct and implement act(belief, t_s, budget_ms)
- rocket/chaser/target attitude_control: must construct and implement act(...)
- bridge (if enabled): must provide at least one of step/start/send_command/receive_command
- mission_objective pointers: must provide a callable function or class method in {evaluate, update, check, act}

Important Current Scope
- The loader validates and parses the YAML into typed config objects.
- Full end-to-end execution from YAML is not yet universal for all scenario runners.
- Use this file now for standardized scenario definitions and input validation.

Recommended Workflow
1. Keep a base scenario YAML.
2. Create variants by changing only metadata + key parameter overrides.
3. Validate with Simulation_Config_Load_Demo.
4. Run your chosen scenario script using parsed config integration as available.
