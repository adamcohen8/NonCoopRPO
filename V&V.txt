Verification and Validation (V&V) Status and Plan
Project: Orbital Engagement Lab
Review Date: 2026-04-03
Basis: repository review plus focused execution spot-checks and MATLAB HPOP cross-validation runs on 2026-04-02 and 2026-04-03

Project Scope Covered
- This project is broader than the earlier ASAT-only framing.
- The current repository supports orbit dynamics, attitude dynamics and control, estimation and object knowledge, rendezvous and engagement scenarios, rocket/ascent components, Monte Carlo and sensitivity analysis, validation tooling, API/GUI workflows, and external integration stubs.

Objective
- Track what has already been verified or partially validated in the codebase.
- Separate implemented V&V infrastructure from still-missing decision-grade evidence.
- Keep a practical roadmap for moving from internal engineering confidence to stronger external validation claims.

Confidence Framework
- Level 0 (Exploratory): trend-only confidence; qualitative insight only.
- Level 1 (Verified Core): core software behavior is regression tested and major numerical/logic paths are exercised.
- Level 2 (Partially Integrated Validated): selected integrated scenarios have benchmark configs, tolerance gates, and repeatable reports.
- Level 3 (Decision Grade): externally anchored truth data, calibrated uncertainty models, and release-quality confidence statements exist for a defined envelope.

Current Assessed Confidence Level
- Current repository evidence supports Level 1 for core software verification.
- There is strong Level 2 evidence for selected orbit force-model validation cases and partial Level 2 evidence for broader integrated validation.
- Level 3 is not achieved in the current repo state.

Evidence Reviewed
- `sim/tests/` currently contains 58 test files and 271 test definitions.
- The repository includes an automated validation harness with YAML specs and JSON/Markdown reports.
- The repository includes HPOP cross-validation tooling for orbit model parity checks.
- The repository includes integrated single-run and Monte Carlo validation scenarios for rendezvous/orbit cases.
- The repository includes sensitivity, Latin hypercube sampling, controller benchmarking, plugin validation, API tests, and GUI/config round-trip tests.

Recommended Current Confidence Statement
- Orbital Engagement Lab is suitable for internal engineering exploration, comparative controller evaluation, and repeatable closed-loop simulation studies.
- The repository now has meaningful internal verification and strong short-horizon orbit force-model validation evidence against MATLAB HPOP for a defined set of tested perturbation cases.
- It is not yet decision-grade for real-world engagement claims without broader external truth comparison, longer-horizon regime coverage, rocket/ascent validation, and formal uncertainty calibration.

Status Against the Earlier Plan

Phase A: Verification (Build Correctness)
1. Deterministic unit tests (Implemented)
- Present across orbit propagation, attitude/control, actuators, rocket stack assembly, mission logic, Monte Carlo reporting, config parsing, API behavior, app I/O, and validation harness helpers.
- Representative coverage exists for J2/J3/J4 and spherical harmonics plugins, atmosphere models, disturbance torques, reaction wheel behavior, orbital actuator mass depletion, mission transitions, and controller bench reporting.
- Gap: no checked-in branch-coverage gate or published critical-path coverage metric was found.

2. Numerical verification (Partial)
- Adaptive orbit integrator selection and RKF78 step behavior are tested.
- Atmosphere, SRP, drag, eclipse, and perturbation helper logic have regression tests.
- HPOP comparison tooling exists and can report RMS/max orbit error metrics for multiple force-model configurations.
- Gap: no systematic dt/tolerance sweep report was found across the full dynamics stack, and event-timing verification is not yet organized as a formal benchmark suite.

3. Interface verification (Partial)
- Scenario YAML parsing, defaults, plugin pointer validation, public API sessions, app config load/save, and app service capabilities are tested.
- Ground-track, frame, and config-section normalization logic are covered by tests.
- cFS SIL bridge files and interface definitions exist under `integrations/cfs_sil/`.
- Gap: no general automated dimensional-consistency checker was found for all interfaces/log outputs.

Phase B: Subsystem Validation (Physics Fidelity)
4. Orbital truth-model validation (Partial)
- `validation/hpop_compare.py` supports model-by-model parity checks for `two_body`, `drag`, `srp`, `j2`, `j3`, `j4`, `j2j3`, `j2j3j4`, and `sh8x8`.
- The automated validation harness includes tolerance tables and HPOP-driven benchmarks.
- The MATLAB HPOP bridge now supports shared configuration-driven runs, HPOP-style DE440 ephemerides on the Python side, and direct cross-validation for isolated and combined perturbation cases.
- Gap: broad regime-by-regime truth matrices are not yet documented, and the validated envelope is still concentrated around short-horizon single-object orbit propagation cases.

5. Attitude and disturbance validation (Partial)
- Disturbance torque modeling, substep behavior, cached environment use, and attitude controller variants are regression tested.
- Reaction wheel and integrated actuator behavior are exercised.
- Guardrail statistics are captured and used in Monte Carlo gating.
- Gap: no external hardware or independent truth-model benchmark package was found for attitude/disturbance fidelity.

6. Rocket/ascent validation (Partial)
- Rocket engine, aero, guidance stack composition, mission split behavior, termination conditions, and representative demos/tests are present.
- Gap: no external trajectory-reference comparison set or ascent-specific tolerance table was found in the validation harness.

7. Sensor/knowledge/estimation validation (Partial, with strong internal checks)
- Object knowledge tracking, line-of-sight/range/solid-angle gating, EKF updates, detection summaries, and consistency summaries are tested.
- Integrated validation configs and harness tolerances already include `nis_mean`, `nees_mean`, update-rate, and position-error checks.
- Gap: no formal Monte Carlo consistency study report was found showing required NIS/NEES bounds across a published envelope.

Phase C: Integrated Scenario Validation
8. Engagement chain validation (Partial)
- Integrated rendezvous configs exist for single-run and Monte Carlo cases.
- Mission executive transitions, integrated burn/attitude sequencing, and controller-to-actuator handoff logic are tested.
- ASAT-style and rendezvous scenario scripts/examples exist.
- Gap: phase-boundary acceptance tables are not yet captured as a formal validation artifact.

9. Adversary behavior validation (Limited)
- The repository includes ASAT phased-engagement scenarios and target-evasion examples.
- Gap: no standardized adversary-behavior library with reproducibility/diversity acceptance criteria was found as part of the V&V workflow.

10. End-to-end benchmark campaigns (Partial)
- Monte Carlo reporting, commander briefs, pass/fail gates, baseline comparison helpers, and confidence-band summaries are implemented.
- The automated validation harness includes integrated rendezvous and high-fidelity orbit Monte Carlo cases.
- Gap: no trusted external integrated baseline campaign appears to be checked in for full end-to-end comparison.

Phase D: Uncertainty Quantification and Confidence Statements
11. Calibration and uncertainty model (Not yet formalized)
- Monte Carlo dispersions and sensitivity parameterization exist.
- Gap: no formal parameter calibration workflow against measured or trusted truth data was found.

12. Monte Carlo confidence characterization (Partial)
- Aggregate pass rates, catastrophic outcome rates, guardrail rates, seed counts, and confidence-style commander-brief summaries are implemented.
- Gap: these outputs are not yet turned into release-quality envelope confidence statements.

13. Global sensitivity analysis (Partial)
- One-at-a-time sensitivity and Latin hypercube sampling are supported through config/API flows.
- Sensitivity rankings are produced from run sets.
- Gap: no standing sensitivity report or model-improvement prioritization process was found.

Phase E: Acceptance and Governance
14. Validation report package (Partial)
- The validation harness writes JSON and Markdown reports.
- Controller benchmarking writes JSON, Markdown, and CSV comparison artifacts.
- Gap: no release report template was found that consistently captures commit hash, environment, dependency versions, assumptions, limitations, and approved envelope statements.

15. Change-control gate (Partial)
- The repo already has regression tests and validation specs that could support a gate.
- Gap: no enforced CI/release gate was found in this review that requires targeted re-validation after dynamics/control/estimation changes.

What Is Still Missing for Stronger Validation Claims
- Broader truth-model matrices across regimes, durations, and perturbation combinations.
- Formal rocket/ascent reference comparisons.
- Formal calibration of uncertain parameters against trusted data.
- Release-grade confidence statements tied to a clearly bounded envelope.
- Documented change-control criteria that make validation reruns mandatory.

Priority Next Steps
1. Expand orbit truth-model validation beyond the current short-horizon matrix.
- Add documented regime tables for LEO, high-LEO, and cislunar cases.
- Add broader force-model combinations and duration sweeps.
- Add Sun-only, Sun+Moon, and higher-altitude third-body cases to the standing benchmark set.

2. Add formal numerical verification campaigns.
- Create automated dt/tolerance convergence sweeps and event-timing checks with stored reports.

3. Promote estimation validation from internal metrics to explicit benchmark evidence.
- Save Monte Carlo NIS/NEES summaries as first-class validation artifacts with pass/fail thresholds by scenario family.

4. Add rocket/ascent external-reference validation.
- Compare altitude, velocity, max-Q timing/value, and burnout conditions against trusted references for at least one representative vehicle profile.

5. Produce a release-grade validation package.
- Include commit hash, Python/package versions, assumptions, known limitations, benchmark matrix, and explicit envelope-of-validity statements.

Decision-Oriented Bottom Line
- The current repo is no longer at the "just build a harness" stage. That work is already substantially done.
- The main gap has shifted from internal verification infrastructure to broader external truth anchoring, wider benchmark coverage, and formal governance.
- The recent HPOP campaign materially improved the project’s V&V position by showing millimeter-level to sub-meter agreement across isolated and combined orbit perturbation cases within the tested envelope.
- If this project remains focused on engagement analysis, the next V&V investment should be external benchmark depth and uncertainty calibration rather than more generic internal plumbing.
